# ADR-0009 — Internationalization: next-intl, cookie-based locale, English default

- Status: **Accepted** (2026-05-08)
- Deciders: founding team
- Tags: i18n, web, verifier, ux, seo
- Related: ADR-0001 (monorepo), CLAUDE.md ("PT-BR product, EN docs")

## Context

YourSign is a PT-BR-first product (per CLAUDE.md) but its public surface is consumed by audiences that expect English: Colosseum judges, recruiters reading the GitHub repo, the international Solana community, and partners outside Brazil. Until now, every UI string in `apps/web` and `apps/verifier` was hardcoded as Portuguese, with `<html lang="pt-BR">` on the root layout. There was no path for an English visitor to understand the product without translating the page in their browser, and no story for adding new locales when product-market fit pulls us into a third language.

User-facing surfaces in scope:

- `apps/web` — the primary web app (landing, `/agents`, `/sign/[hash]`, components).
- `apps/verifier` — the public verifier (`verify.yoursign.tech`).

User-facing surfaces explicitly **out of scope** for this ADR:

- `apps/api`, `apps/mcp`, `apps/worker` — return JSON to programmatic clients; their error strings are developer-facing identifiers, not UI copy.
- `programs/yoursign/**` — on-chain program; no human strings.
- `packages/ui/src/scope-preview.tsx` — currently unimported from any app; will be revisited the next time it's actually consumed.

## Options considered

1. **`next-intl` v4 with cookie-based locale resolution (this ADR).**
2. **`next-intl` v4 with `[locale]` route segments** (`/en/...`, `/pt/...` or `/agents` + `/en/agents`).
3. **`react-i18next`** — the React-ecosystem default; no first-class App Router server-component support, requires `'use client'` plumbing on every translated subtree.
4. **Lingui (`@lingui/react` + `@lingui/macro`).** Compiles messages, AST-level extraction. Strong DX but heavier toolchain, separate compile step, smaller community on Next App Router.
5. **Paraglide JS (Inlang).** Ultra-light runtime, tree-shakable per-message. Great fit for static export, but its App Router story is still maturing and it requires a custom Vite/Turbopack pipeline we'd own.
6. **Hand-rolled `Intl` + JSON dictionaries.** Cheapest. No `ICU` plural support, no message extraction tooling, no formatting primitives. Sufficient for v0 but leaves real work on the table for the next locale.

URL strategy sub-options for next-intl:

- **Cookie / `Accept-Language`** — single canonical URL per page; locale is a side-channel.
- **Route prefix, default visible** — `/pt/agents`, `/en/agents`. Most explicit, breaks every existing inbound link.
- **Route prefix, default hidden** — `/agents` (default) and `/en/agents` (other). SEO-friendly but adds a `[locale]` segment to every route.

## Decision

- **Library:** `next-intl@^4.3.4`, wired via `createNextIntlPlugin('./i18n/request.ts')` in each app's `next.config.ts`.
- **Locale resolution:** server-side, in this priority order:
  1. `NEXT_LOCALE` cookie (long-lived, `Max-Age = 1y`, `SameSite=Lax`).
  2. `Accept-Language` header — if it starts with `pt`, locale becomes `pt`.
  3. Default: `en`.
- **URL strategy:** cookie-based. URLs do not change between locales. No `[locale]` route segment.
- **Default locale:** `en`. PT-BR speakers get auto-detected via `Accept-Language` on first hit, then persisted via cookie when they explicitly switch.
- **Locale switcher:** a single `<select>` rendered in each app's nav. It calls a server action (`app/_actions/set-locale.ts`) that writes the cookie and `revalidatePath('/', 'layout')`s the root.
- **Message files:** colocated per app under `messages/{en,pt}.json`. Hierarchical keys (`home.how.step1.title`) — no flat dot-files. ICU plurals where needed (e.g. `agents.scopePreview.docsHashes`).
- **Server vs client:** server components use `getTranslations()` / `getLocale()` from `next-intl/server`; client components use `useTranslations()` / `useLocale()`. Messages are bridged once via `<NextIntlClientProvider>` in the root layout.
- **`<html lang>`:** dynamic, set from the resolved locale in the root server layout.
- **Number / date formatting:** caller-side `toLocaleString(locale === 'pt' ? 'pt-BR' : 'en-US', …)` for now. Will move to next-intl's `formats` config when we add a third locale.
- **Supported locales constant:** `apps/{web,verifier}/i18n/locales.ts` — single source of truth for `SUPPORTED_LOCALES`, `DEFAULT_LOCALE`, `LOCALE_LABELS`. Both apps duplicate this file deliberately (no shared `packages/i18n` until a third consumer asks for it — see CLAUDE.md "no premature abstraction").

## Why

- **next-intl v4 is the canonical Next 16 App Router answer.** Server-component-first, official integration, Vercel-blessed. Picking the default keeps the harness boring and predictable.
- **Cookie strategy avoids restructuring routes.** No `[locale]` segment means no rewrite of `app/sign/[hash]/page.tsx`, no broken inbound links to `/agents`, no SEO hreflang spreadsheet for v0. Trade-off accepted: see Consequences.
- **English default reaches the international audience first.** The PT-BR speaker cost is one extra hop (cookie write on first switch) — the first-visit experience is still PT-BR for any browser that asks for it via `Accept-Language`.
- **Server-side resolution makes `<html lang>` correct on first paint.** No flash of wrong language, no client-side hydration mismatch.
- **ICU message format ships with next-intl.** Plurals (`{count, plural, ...}`) and rich text (`t.rich`) work out of the box — no extra plugin.

## Why NOT the alternatives

- **Route prefix (any flavor).** Every existing inbound link to `/agents` or `/sign/<hash>` would need a redirect rule. SEO benefit is real but the product surface is too small (3 pages × 2 locales) to justify the URL churn at this stage. We can revisit when SEO is a measurable goal.
- **`react-i18next`.** No native App Router server-component story; would force `'use client'` boundaries to push down to every translated leaf. Loses RSC payload benefits.
- **Lingui.** Strong DX but the macro-based extraction toolchain is heavier than what 2 apps + 2 locales need today. Reconsider if message volume crosses ~500.
- **Paraglide JS.** Promising but its Next App Router integration is less mature; adopting it would mean owning bug reports against an immature stack. Not the time.
- **Hand-rolled.** No ICU, no formatter primitives, no extraction tooling. Saves one dependency but loses the entire ecosystem the next time we need plurals or relative dates. Net negative.
- **Cookie-less URLs forever.** Acceptable for v0; we're not blocking ourselves from adding `[locale]` later — `next-intl` supports both modes from the same config surface.

## Consequences

- **Dynamic rendering by default.** Reading `cookies()` and `headers()` in the layout opts the root out of static rendering. For our app — wallet integration, dynamic server actions — this was already the reality. The verifier was theoretically static; it now is not. Acceptable: a 2KB JSON read on Cloudflare Workers is cheap.
- **Cloudflare Workers compat.** `cookies()` from `next/headers` is supported by `@opennextjs/cloudflare`. No new infra.
- **Cookie cold-start UX.** A first-time PT-BR visitor with `Accept-Language: pt-BR,*;q=0.5` lands on Portuguese as expected. A user who landed on English and wants Portuguese must click the switcher once; the choice persists for a year.
- **CDN cache key.** Cookie-driven content means we should add `Vary: Cookie` if we ever cache HTML at the edge. Today we don't; if we do, the deployment runbook (Sprint 4) needs an entry.
- **SEO.** `<html lang>` is correct per request, but a single canonical URL serves both languages — Google sees one page with mixed signals. If organic search becomes a goal, switch to route-prefixed locales (path already laid by `next-intl`).
- **Bundle size.** `next-intl` adds ~12KB gzipped to the client runtime per app. Acceptable.
- **Translation pipeline.** Manual for now: every PR that adds copy must touch both `en.json` and `pt.json`. We'll add a verifier (`harness/verifiers/i18n-key-parity.md`) once a missed key bites us.
- **Tests / typecheck.** No runtime test coverage added — `pnpm typecheck` and `pnpm test` both green on the feature branch. Translation key correctness is currently checked by typecheck only via `next-intl`'s type narrowing on call sites.
- **CLAUDE.md alignment.** Product remains PT-BR-first in product copy quality; English is a translation, not the source. Documentation, ADRs, contracts, and code identifiers stay English (already the rule).

## Reversal cost

- **To switch URL strategy** (cookie → route prefix): touch each app's `i18n/request.ts` to read the segment, add a `middleware.ts`, move `app/*` under `app/[locale]/*`. ~half a day per app. No data migration.
- **To switch library** (next-intl → other): the `messages/*.json` files are mostly portable; ICU is widely supported. The biggest hit is the call-site refactor (`useTranslations` → equivalent). ~1 day per app.
- **To drop i18n entirely** (revert to PT-BR-only): revert this commit. The branch is one squash-merge away from a rollback.

## Implementation pointers

- `apps/{web,verifier}/i18n/request.ts` — locale resolution.
- `apps/{web,verifier}/i18n/locales.ts` — supported locales constants.
- `apps/{web,verifier}/messages/{en,pt}.json` — translations.
- `apps/{web,verifier}/app/_actions/set-locale.ts` — cookie-writing server action.
- `apps/{web,verifier}/app/components/language-switcher.tsx` (web) / `app/components/language-switcher.tsx` (verifier) — client switcher.
- `apps/{web,verifier}/app/layout.tsx` — `NextIntlClientProvider` + `<html lang>`.
- `apps/{web,verifier}/next.config.ts` — `createNextIntlPlugin('./i18n/request.ts')`.
